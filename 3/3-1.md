# 3-1. 기본 Git 명령

## Claude Code의 Git 통합 개요

Claude Code는 Git을 직접 이해하고 자연어 요청을 적절한 Git 명령으로 변환해 실행합니다. 단순한 명령 래핑이 아니라, 변경사항의 의미를 파악해 적절한 커밋 메시지 작성, 충돌 분석, 히스토리 해석까지 수행합니다.

**Claude Code의 Git 안전 원칙:**
- 파괴적 명령(`reset --hard`, `push --force`, `clean -f` 등)은 사용자가 명시적으로 요청할 때만 실행
- 후크(pre-commit, pre-push 등)를 `--no-verify`로 우회하지 않음
- 커밋 전 변경사항을 먼저 보여주고 확인 요청
- main/master 브랜치에 force push 시 경고

## 자연어로 Git 명령 실행

### 변경사항 확인

```
> 변경된 파일 보여줘
```
→ `git status` 실행 후 결과를 사람이 이해하기 쉬운 형태로 설명

```
> 뭘 수정했는지 보여줘
> 스테이징된 파일과 안 된 파일 구분해서 알려줘
```

### 커밋 히스토리 조회

```
> 마지막 5개 커밋을 보여줘
> 오늘 작업한 커밋들을 보여줘
> 이 파일의 변경 이력을 알려줘
```

Claude가 내부적으로:
```bash
git log --oneline -5
git log --since="today" --format="%h %s %an"
git log --follow -p src/auth.py
```

### diff 분석

```
> 이 변경사항이 뭔지 설명해줘
> main 브랜치와 지금 브랜치의 차이를 설명해줘
> 마지막 커밋에서 뭘 바꿨어?
```

Claude가:
```bash
git diff
git diff main...HEAD
git show HEAD
```

결과를 단순 출력이 아니라 **의미 있는 설명**으로 제공:
```
지난 커밋에서 세 가지 변경이 있었습니다:
1. src/auth.py: JWT 토큰 만료 시간을 1시간에서 24시간으로 변경
2. src/models/user.py: last_login 필드 추가
3. tests/test_auth.py: 토큰 만료 테스트 케이스 3개 추가
```

## 커밋 작업

### 커밋 메시지 자동 생성

```
> 지금 변경사항으로 커밋해줘
> 설명적인 메시지로 커밋해줘
> 커밋 메시지 제안해줘
```

Claude가 `git diff --staged`를 분석하고 적절한 메시지를 생성합니다:

```bash
# Claude가 생성하는 커밋 예시
git commit -m "feat: JWT 토큰 만료 시간을 24시간으로 변경

- 기존 1시간에서 24시간으로 변경하여 사용자 편의성 향상
- last_login 필드를 User 모델에 추가
- 토큰 만료 관련 테스트 케이스 보강"
```

### 특정 파일만 커밋

```
> auth.py만 스테이징해서 커밋해줘
> src 폴더의 변경사항만 커밋해줘
> 테스트 파일은 빼고 커밋해줘
```

```bash
git add src/auth.py
git commit -m "fix: 인증 토큰 갱신 로직 수정"
```

### 커밋 타입 지정

```
> 버그 수정으로 커밋해줘 (fix 타입)
> 새 기능으로 커밋해줘 (feat 타입)
> 리팩토링으로 커밋해줘 (refactor 타입)
```

Conventional Commits 형식 지원:
- `feat`: 새 기능
- `fix`: 버그 수정
- `docs`: 문서
- `style`: 코드 스타일
- `refactor`: 리팩토링
- `test`: 테스트
- `chore`: 빌드, 설정

## 변경사항 분석

### 무엇이 바뀌었나 파악

```
> 이번 주에 가장 많이 수정된 파일이 뭐야?
> 내가 마지막으로 이 파일을 수정한 게 언제야?
> 이 함수가 언제 추가됐어?
```

Claude가 `git log`, `git blame`, `git log -S` 등을 조합해 분석합니다.

### 브랜치 간 비교

```
> main이랑 현재 브랜치 차이를 요약해줘
> 이 PR에서 변경된 파일 목록을 보여줘
```

```bash
git diff main...HEAD --name-only
git diff main...HEAD --stat
```

## 충돌 해결

```
> 머지 충돌이 있어. 해결 도와줘.
> 어떤 파일에서 충돌이 났어?
> 이 충돌을 어떻게 해결해야 해?
```

Claude가:
1. `git status`로 충돌 파일 목록 파악
2. 각 충돌 파일 Read로 내용 확인
3. `<<<<<<<`, `=======`, `>>>>>>>` 마커 분석
4. 의도에 맞는 해결책 제안 또는 적용

충돌 해결 예시:
```
> @src/models/user.py 의 충돌을 해결해줘. main 브랜치 변경사항을 유지하되,
  내 브랜치에서 추가한 avatar_url 필드는 반드시 포함해줘.
```

## 되돌리기 작업

### 마지막 커밋 취소

```
> 마지막 커밋을 취소해줘 (변경사항은 유지)
```

Claude가:
```bash
git reset HEAD~1
# (파일은 변경된 상태로 유지, 커밋만 취소)
```

주의: `--hard` 옵션은 명시적으로 요청해야만 사용합니다.

```
> 마지막 커밋을 완전히 지워줘 (변경사항도 삭제)
Claude: 주의 - 이 작업은 변경사항을 완전히 삭제합니다.
        정말 git reset --hard HEAD~1을 실행할까요?
```

### 특정 파일 되돌리기

```
> src/auth.py를 마지막 커밋 상태로 되돌려줘
```

```bash
git checkout HEAD -- src/auth.py
```

## `--no-verify` 금지 원칙

Claude Code는 Git 후크를 절대 우회하지 않습니다.

```
> 후크 무시하고 빨리 커밋해줘
Claude: 죄송합니다. --no-verify는 사용하지 않습니다.
        pre-commit 후크가 실패했다면, 그 원인을 먼저 해결해야 합니다.
        어떤 오류가 발생했는지 알려주시면 도움드리겠습니다.
```

후크 실패 시 올바른 접근:
```
> pre-commit 후크가 실패했어. 원인을 찾아서 고쳐줘.
```

## 실습

### 실습 1: 변경사항 분석

```bash
cd ~/my-project
# 파일 몇 개 수정 후
claude
```

```
> 변경된 파일들을 분석해서 무엇을 수정했는지 설명해줘.
  그리고 커밋 메시지를 제안해줘.
```

### 실습 2: 히스토리 분석

```
> 이 프로젝트의 마지막 10개 커밋을 분석해서
  어떤 작업이 주로 이루어졌는지 요약해줘.
```

### 실습 3: 커밋 메시지 생성

```bash
# 변경사항 스테이징
git add src/

# Claude에게 커밋 요청
```

```
> 스테이징된 변경사항으로 Conventional Commits 형식의 커밋 메시지를 만들어줘.
  영어와 한국어 버전 둘 다 제안해줘.
```

### 실습 4: diff 분석

```
> git diff HEAD~3..HEAD 결과를 분석해서
  지난 3개 커밋에서 어떤 변화가 있었는지 설명해줘.
```
