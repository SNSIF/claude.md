# 5-1. CLAUDE.md 작성

CLAUDE.md는 Claude에게 프로젝트 컨텍스트와 행동 규칙을 알려주는 핵심 파일입니다. Claude Code는 세션 시작 시 이 파일을 자동으로 읽어 "이 프로젝트에서 어떻게 행동해야 하는가"를 파악합니다.

---

## CLAUDE.md의 역할

CLAUDE.md가 없으면 Claude는 프로젝트에 대해 아무것도 모른 채 시작합니다. 빌드 명령어가 뭔지, 코드 스타일이 어떤지, 어떤 아키텍처 결정이 내려졌는지를 매번 설명해야 합니다.

CLAUDE.md가 있으면:

- 빌드·테스트 명령어를 매번 물어보지 않습니다
- 프로젝트의 코딩 컨벤션을 자동으로 따릅니다
- 아키텍처 결정의 배경을 알고 코드를 작성합니다
- 팀의 금지 사항(특정 라이브러리 미사용 등)을 지킵니다

---

## 파일 위치 우선순위

여러 위치의 CLAUDE.md가 순서대로 로드되며 모두 적용됩니다.

```
1. ~/.claude/CLAUDE.md             ← 사용자 전역 (모든 프로젝트 공통 규칙)
2. ./CLAUDE.md                     ← 프로젝트 루트
   또는 ./.claude/CLAUDE.md
3. 각 서브디렉토리의 CLAUDE.md      ← 해당 경로 작업 시 추가 로드
```

우선순위는 아래로 갈수록 높습니다. 더 구체적인 위치의 규칙이 상위를 재정의합니다.

### 전역 CLAUDE.md 예시 (`~/.claude/CLAUDE.md`)

```markdown
# 개인 공통 규칙

## 코드 스타일
- 모든 프로젝트에서 2칸 들여쓰기 사용
- 변수명은 camelCase, 상수는 UPPER_SNAKE_CASE

## 커밋 규칙
- Conventional Commits 형식 사용
- 한국어 커밋 메시지 허용

## 응답 규칙
- 코드 설명은 한국어로
- 코드 자체는 영어 식별자 사용
```

---

## `/init` 커맨드로 자동 생성

새 프로젝트에서 CLAUDE.md를 처음 만들 때는 `/init`을 사용합니다.

```
/init
```

Claude가 현재 디렉토리의 파일 구조, `package.json`, `pyproject.toml` 등을 분석해 기본 CLAUDE.md를 자동 생성합니다.

자동 생성 후 반드시 내용을 검토하고 팀 규칙에 맞게 수정하세요.

## 잘 쓴 CLAUDE.md 예시

아래는 실제 프로젝트에 적용할 수 있는 완성도 높은 예시입니다.

```markdown
# my-saas-project

SaaS 구독 관리 서비스. React + FastAPI + PostgreSQL 스택.

## 빌드 명령어

### 프론트엔드
- 개발 서버: `npm run dev`
- 빌드: `npm run build`
- 테스트: `npm test`
- 린트: `npm run lint`

### 백엔드
- 개발 서버: `uvicorn app.main:app --reload`
- 테스트: `pytest -v`
- 마이그레이션: `alembic upgrade head`

## 코드 스타일

- 들여쓰기: 2칸 (프론트엔드), 4칸 (백엔드)
- TypeScript strict 모드 활성화 — `any` 타입 금지
- Python: Black 포매터 사용, 줄 길이 88자
- 컴포넌트는 함수형으로만 작성 (클래스 컴포넌트 금지)

## 아키텍처

### 프론트엔드
- 라우터가 가리키는 페이지: `src/container/`
- 재사용 UI: `src/component/`
- API 통신: TanStack Query (axios 직접 호출 금지)
- 전역 상태: Context API만 사용 (Redux 미사용)

### 백엔드
- 기능별 모듈 구조: `app/module/<기능>/`
- 각 모듈: `router.py` / `service.py` / `repository.py`
- 비즈니스 로직은 service, DB 쿼리는 repository에만

## 환경 설정

- 환경 변수는 `.env` 파일에 (절대 하드코딩 금지)
- `.env.example`을 항상 최신 상태로 유지
- 시크릿 키는 절대 커밋하지 않는다

## 금지 사항

- `moment.js` 미사용 → `date-fns` 사용
- `var` 선언 금지 → `const`/`let` 사용
- `console.log` 프로덕션 코드에 남기지 않기
- 직접 DOM 조작 금지 (React ref 통해서만)

## 테스트 규칙

- 새 기능에는 반드시 테스트 작성
- 커버리지 80% 이상 유지
- 테스트 파일명: `*.test.ts` (프론트), `test_*.py` (백엔드)
```

## `@파일명` 임포트 문법

CLAUDE.md 내에서 다른 파일의 내용을 참조할 수 있습니다.

```markdown
# 프로젝트명

@./docs/architecture.md
@./docs/api-conventions.md
```

Claude는 세션 시작 시 이 파일들을 함께 읽어 컨텍스트에 포함합니다. 긴 문서를 CLAUDE.md에 직접 쓰는 대신 별도 파일로 분리하고 임포트하면 관리가 편해집니다.

```markdown
# CLAUDE.md

## 빌드 명령어
- dev: `npm run dev`
- test: `npm test`

## 아키텍처 상세
@./docs/architecture.md

## API 규약
@./docs/api-spec.md

## 팀 코딩 규칙
@./.eslintrc.md
```

## 경로별 규칙 (frontmatter)

특정 디렉토리에만 적용되는 규칙을 서브디렉토리의 CLAUDE.md로 분리할 수 있습니다.

```
프로젝트 루트/
├── CLAUDE.md               ← 전체 프로젝트 규칙
├── src/
│   └── api/
│       └── CLAUDE.md       ← API 디렉토리 전용 규칙
└── tests/
    └── CLAUDE.md           ← 테스트 디렉토리 전용 규칙
```

frontmatter의 `paths`로 적용 범위를 명시할 수도 있습니다.

```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API 파일 전용 규칙

- 모든 엔드포인트 함수에 JSDoc 주석 필수
- 응답 타입은 반드시 명시적으로 선언
- HTTP 상태 코드는 상수로 정의 (예: `HTTP_200_OK`)
- 에러 응답은 `{ error: string, code: string }` 형식 통일
```

```markdown
---
paths:
  - "tests/**"
  - "**/*.test.ts"
---

# 테스트 파일 전용 규칙

- 테스트 설명은 한국어로 작성
- `describe` → `context` 별칭 사용 가능
- 각 테스트는 독립적으로 실행 가능해야 함
- beforeEach에서 상태 초기화 필수
```

## 200줄 이하 유지 권장

CLAUDE.md가 너무 길면 컨텍스트를 불필요하게 소모합니다. 200줄 이하를 목표로 하고, 긴 내용은 `@임포트`로 분리합니다.

| 항목 | 권장 처리 |
|------|-----------|
| 빌드/테스트 명령어 | CLAUDE.md에 직접 포함 |
| 아키텍처 설명 | 별도 `docs/architecture.md` + `@임포트` |
| API 명세 | Swagger 파일 경로만 명시 |
| 긴 코딩 규칙 | ESLint 설정 파일 참조 |

## 실습: 현재 프로젝트용 CLAUDE.md 작성

### 1단계: `/init`으로 초안 생성

```bash
cd ~/my-project
claude
/init
```

### 2단계: 생성된 파일 검토

```bash
# 생성된 내용 확인
cat CLAUDE.md
```

### 3단계: 팀 규칙 추가

생성된 파일을 열어 아래 항목을 추가합니다.

```markdown
## 금지 사항
- [ 프로젝트에서 사용하지 않는 라이브러리 목록 ]

## 코딩 규칙
- [ 팀 컨벤션 ]

## 주의 파일
- `src/core/auth.ts` — 인증 로직 핵심 파일, 수정 전 팀 리뷰 필수
```

### 4단계: 효과 확인

```bash
# 새 세션에서 확인
/clear
빌드 명령어가 뭐야?

# CLAUDE.md의 내용을 바탕으로 즉시 답변하는지 확인
```
