# 4-2. 커스텀 스킬 만들기

스킬(Skill)은 반복적으로 수행하는 워크플로우를 Markdown 파일로 정의해두는 기능입니다. 매번 긴 지시를 타이핑하는 대신 `/스킬명` 하나로 복잡한 작업 절차를 실행할 수 있습니다.

---

## 스킬 개념

스킬은 다음 두 가지 측면에서 일반 CLAUDE.md 지침과 다릅니다.

| 구분 | CLAUDE.md | 스킬 |
|------|-----------|------|
| 목적 | 항상 적용되는 프로젝트 규칙 | 필요할 때 호출하는 절차 |
| 활성화 | 자동 (매 세션 로드) | 명시적 호출 또는 자동 위임 |
| 형식 | 자유 Markdown | frontmatter + 절차 |

스킬 파일 한 개가 하나의 슬래시 커맨드가 됩니다. `deploy.md`를 만들면 `/deploy`로 호출합니다.

---

## 파일 위치

### 전역 스킬 — 모든 프로젝트에서 사용

```
~/.claude/skills/
├── commit.md
├── deploy.md
└── review.md
```

어디서 `claude`를 실행하든 이 스킬들을 사용할 수 있습니다.

### 프로젝트 스킬 — 해당 프로젝트 전용

```
프로젝트 루트/
└── .claude/
    └── skills/
        ├── seed-db.md
        └── gen-migration.md
```

프로젝트 디렉토리 안에서만 적용됩니다. 같은 이름의 스킬이 전역과 프로젝트에 모두 있으면 프로젝트 스킬이 우선합니다.

---

## 스킬 파일 구조

스킬 파일은 YAML frontmatter와 Markdown 본문으로 구성됩니다.

```markdown
---
name: deploy
description: 애플리케이션을 스테이징 환경에 배포합니다
allowed-tools: Bash, Read
---

## 배포 전 검증

1. 현재 브랜치가 `main`인지 확인한다
2. 미커밋 변경사항이 없는지 확인한다
3. 테스트를 실행한다: `npm test`

## 빌드

4. 프로덕션 빌드를 생성한다: `npm run build`
5. 빌드 결과물 크기를 확인한다

## 배포

6. 스테이징 서버에 배포한다: `npm run deploy:staging`
7. 헬스체크 엔드포인트를 호출해 응답을 확인한다
8. 배포 결과를 요약해서 보고한다
```

---

## frontmatter 옵션 상세

### `name`

슬래시 커맨드로 사용할 이름입니다. 파일명과 다르게 지정할 수 있습니다.

```yaml
---
name: db-migrate
---
```

`/db-migrate`로 호출됩니다. 지정하지 않으면 파일명(확장자 제외)이 커맨드명이 됩니다.

---

### `description`

스킬의 목적을 한 문장으로 설명합니다. Claude가 대화 내용을 분석해서 적합한 스킬을 자동 선택(위임)할 때 이 설명을 기준으로 판단합니다.

```yaml
---
description: Git 커밋 메시지를 작성하고 스테이징된 변경사항을 커밋합니다
---
```

설명이 구체적일수록 자동 위임 정확도가 높아집니다.

---

### `allowed-tools`

스킬 실행 중 사용할 수 있는 도구를 제한합니다. 쉼표로 구분하여 나열합니다.

```yaml
---
allowed-tools: Bash, Read, Grep
---
```

사용 가능한 도구 이름:

| 도구 | 기능 |
|------|------|
| `Bash` | 셸 명령 실행 |
| `Read` | 파일 읽기 |
| `Write` | 파일 쓰기 |
| `Edit` | 파일 편집 |
| `Glob` | 파일 검색 |
| `Grep` | 내용 검색 |
| `WebFetch` | URL 내용 가져오기 |
| `WebSearch` | 웹 검색 |

지정하지 않으면 settings.json의 권한 설정을 따릅니다.

---

### `disable-model-invocation`

`true`로 설정하면 Claude 모델 없이 스킬 본문의 명령을 직접 실행합니다. 단순한 셸 스크립트처럼 동작합니다.

```yaml
---
disable-model-invocation: true
allowed-tools: Bash
---

다음 명령을 실행한다:
- `git status`
- `git diff --stat`
```

이 경우 스킬 본문은 자연어 지시가 아닌 직접 실행 가능한 명령 목록이어야 합니다.

---

## 스킬 내 변수 활용

스킬 호출 시 인자(argument)를 전달하면 `$ARGUMENTS` 변수로 참조할 수 있습니다.

```markdown
---
name: pr-review
description: 지정한 PR 번호의 코드를 리뷰합니다
allowed-tools: Bash, WebFetch
---

PR #$ARGUMENTS 를 리뷰한다.

1. `gh pr view $ARGUMENTS` 로 PR 정보를 가져온다
2. `gh pr diff $ARGUMENTS` 로 변경 코드를 확인한다
3. 변경 범위, 잠재적 버그, 개선점을 분석한다
4. 리뷰 코멘트를 작성한다
```

호출 방법:

```
/pr-review 142
```

`$ARGUMENTS`에 `142`가 대입되어 PR #142를 리뷰합니다.

## 실습: commit 자동화 스킬 만들기

커밋 메시지를 자동 생성하고 커밋하는 스킬을 단계별로 만듭니다.

### 1단계: 스킬 파일 생성

```bash
mkdir -p ~/.claude/skills
```

아래 내용으로 `~/.claude/skills/commit.md`를 생성합니다.

```markdown
---
name: commit
description: 스테이징된 변경사항을 분석해 커밋 메시지를 생성하고 커밋합니다
allowed-tools: Bash
---

## 사전 확인

1. `git status` 로 현재 상태를 확인한다
2. 스테이징된 파일이 없으면 "스테이징된 변경사항이 없습니다"라고 알리고 중단한다

## 변경사항 분석

3. `git diff --staged` 로 스테이징된 내용을 읽는다
4. 변경 내용을 다음 기준으로 분류한다:
   - feat: 새로운 기능 추가
   - fix: 버그 수정
   - refactor: 코드 구조 개선
   - docs: 문서 변경
   - test: 테스트 추가/수정
   - chore: 빌드, 설정 파일 변경

## 커밋 메시지 작성

5. 형식: `타입: 한 줄 요약 (50자 이하)`
6. 변경이 복잡하면 본문에 상세 설명 추가
7. 작성한 커밋 메시지를 사용자에게 보여주고 확인을 받는다

## 커밋 실행

8. 승인 받으면 `git commit -m "메시지"` 를 실행한다
9. 커밋 해시와 함께 완료 메시지를 출력한다
```

### 2단계: 스킬 등록 확인

```
/skills
```

`commit` 스킬이 목록에 표시되면 성공입니다.

### 3단계: 스킬 사용

```bash
# 먼저 변경사항 스테이징
git add src/auth/login.ts

# 스킬 실행
/commit
```

Claude가 `git diff --staged`를 읽고 커밋 메시지를 제안합니다.

```
분석 완료. 커밋 메시지 제안:

  feat: JWT 토큰 만료 시간 설정 기능 추가

커밋 본문:
  - 환경 변수 JWT_EXPIRES_IN으로 만료 시간 설정 가능
  - 기본값 24h 적용

이 메시지로 커밋할까요?
```

승인하면 커밋이 실행됩니다.

### 4단계: 발전시키기

스킬을 더 정교하게 만드는 예시입니다.

```markdown
---
name: commit
description: Conventional Commits 형식으로 커밋합니다
allowed-tools: Bash
---

## 사전 확인

1. `git status` 확인
2. 스테이징 없으면 중단

## 브랜치 정보 수집

3. `git branch --show-current` 로 현재 브랜치를 확인한다
4. 브랜치명에서 이슈 번호를 추출한다 (예: feature/AUTH-123 → AUTH-123)

## 변경사항 분석

5. `git diff --staged --stat` 로 파일 목록 확인
6. `git diff --staged` 로 실제 변경 내용 확인

## 커밋 메시지 생성

7. Conventional Commits 형식으로 작성:
   ```
   타입(스코프): 요약

   본문 (선택)

   Refs: #이슈번호
   ```
8. 사용자에게 확인 요청

## 실행

9. 확인 받으면 커밋 실행
10. `git log --oneline -1` 로 커밋 결과 출력
```
