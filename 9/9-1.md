# 9-1. 내장 에이전트 (Built-in Sub-Agents)

Claude Code는 단일 인스턴스로만 동작하는 것이 아니라, 필요에 따라 **서브에이전트(Sub-Agent)**를 생성하여 작업을 위임합니다. 내장 에이전트는 Claude Code에 기본으로 탑재된 특수 목적의 하위 에이전트입니다.

---

## 서브에이전트(Sub-Agent) 개념

서브에이전트란 **독립적인 컨텍스트에서 작업을 수행하는 하위 Claude 인스턴스**입니다. 메인 에이전트(사용자와 대화하는 Claude)가 특정 작업을 별도의 Claude 인스턴스에게 위임하는 구조입니다.

```
사용자
  ↕
메인 에이전트 (Claude Sonnet)
  ├── 서브에이전트 A (탐색 작업)
  ├── 서브에이전트 B (코드 분석)
  └── 서브에이전트 C (계획 수립)
```

서브에이전트는 메인 에이전트와 **별도의 컨텍스트 창**을 사용합니다. 즉, 서브에이전트가 수행한 탐색 내용이 메인 에이전트의 토큰 예산을 소모하지 않습니다.

---

## 메인 에이전트가 서브에이전트에 작업을 위임하는 이유

### 1. 메인 컨텍스트 보호 (토큰 절약)

코드베이스 전체를 탐색하면 수만 개의 토큰이 소모됩니다. 서브에이전트가 탐색을 수행하고 **요약된 결과**만 메인 에이전트에게 반환하면, 메인 컨텍스트를 깨끗하게 유지할 수 있습니다.

```
[서브에이전트 탐색]
  → 1,000개 파일 탐색 (서브에이전트 컨텍스트 소모)
  → 핵심 결과 3줄 요약 (메인 컨텍스트에 전달)

결과: 메인 에이전트는 3줄만 소비
```

### 2. 병렬 처리 가능

여러 서브에이전트를 동시에 실행하면 작업 시간을 단축할 수 있습니다.

```
메인 에이전트
  ├── [병렬] 서브에이전트 A: frontend/ 탐색
  ├── [병렬] 서브에이전트 B: backend/ 탐색
  └── [병렬] 서브에이전트 C: 테스트 파일 탐색

→ 3개 동시 실행 → 결과 통합
```

### 3. 권한 분리

서브에이전트마다 다른 권한 모드를 부여할 수 있습니다.

```
메인 에이전트: bypassPermissions (자동화)
  ├── 탐색 서브에이전트: plan 모드 (읽기만)
  └── 편집 서브에이전트: acceptEdits 모드
```

## 내장 서브에이전트 3종 상세

### Explore (탐색 전용)

| 속성 | 내용 |
|------|------|
| 모델 | Claude Haiku (빠르고 저렴) |
| 허용 도구 | Read, Grep, Glob만 |
| 금지 도구 | Write, Edit, Bash 등 쓰기 도구 전부 |
| 주요 용도 | 코드베이스 탐색, 파일 검색, 패턴 분석 |

**특징**:
- Haiku 모델 사용으로 빠른 탐색 속도와 낮은 비용
- 읽기 전용이므로 어떤 코드베이스에서도 안전하게 사용 가능
- 메인 에이전트의 지시를 받아 넓은 범위를 탐색하고 결과만 반환

**자동 활용 예시**:
```
사용자: "이 코드베이스가 어떻게 구성돼 있어?"
         ↓
메인 에이전트: "Explore 에이전트로 전체 구조 파악"
         ↓
Explore 에이전트: (자동 실행)
  - Glob으로 파일 트리 파악
  - 핵심 파일들 Read로 내용 확인
  - Grep으로 패턴 탐색
         ↓
결과 요약을 메인 에이전트에 반환
```

**자동으로 Explore가 호출되는 상황**:
- "이 프로젝트 구조 파악해줘"
- "버그 원인을 찾아줘" (먼저 탐색 필요)
- "어떤 파일들이 인증을 담당하고 있어?"
- "의존성이 어떻게 연결돼 있어?"

### Plan (계획 수립)

| 속성 | 내용 |
|------|------|
| 모델 | Claude Sonnet |
| 허용 도구 | Read, Grep, Glob (읽기 전용) |
| 금지 도구 | Write, Edit, Bash |
| 주요 용도 | 구현 전 계획 초안 작성, 설계 검토 |

**특징**:
- Plan Mode(`--permission-mode plan`)와 연계되어 동작
- 코드베이스를 읽고 구체적인 구현 계획을 수립
- 계획이 확정되면 메인 에이전트 또는 다른 에이전트가 실행

**자동 활용 예시**:
```
사용자: "새 결제 시스템 추가하기 전에 계획을 먼저 세워줘"
         ↓
Plan 에이전트: (자동 실행)
  - 기존 결제 관련 코드 탐색
  - 연관 모듈 파악
  - 영향 범위 분석
         ↓
구현 계획서 출력:
  Phase 1: PaymentService 클래스 생성
  Phase 2: /api/payments 라우터 추가
  Phase 3: 프론트엔드 결제 폼 연동
  ...
```

### general-purpose (범용)

| 속성 | 내용 |
|------|------|
| 모델 | Claude Sonnet (메인과 동일) |
| 허용 도구 | 모든 도구 (Read, Write, Edit, Bash 등) |
| 금지 도구 | 없음 |
| 주요 용도 | 복잡한 멀티스텝 작업, 독립 실행 태스크 |

**특징**:
- 메인 Claude와 동일한 능력을 갖춘 완전한 에이전트
- 완전히 독립적인 컨텍스트에서 긴 작업을 수행
- 결과만 메인 에이전트에게 보고

**자동 활용 예시**:
```
사용자: "로그인 기능을 처음부터 끝까지 구현해줘"
         ↓
general-purpose 에이전트:
  Step 1: 기존 코드 탐색 (Glob, Read)
  Step 2: DB 스키마 파일 수정 (Edit)
  Step 3: 백엔드 API 구현 (Write, Edit)
  Step 4: 프론트엔드 컴포넌트 구현 (Write, Edit)
  Step 5: 테스트 실행 (Bash)
         ↓
"로그인 기능 구현 완료" 결과 보고
```

## 서브에이전트 작동 흐름

```
┌─────────────────────────────────────────────┐
│              사용자 요청                      │
│   "auth 모듈 전체를 리팩토링해줘"             │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│         메인 에이전트 (Orchestrator)          │
│                                             │
│  1. 요청 분석                                │
│  2. 서브에이전트 선택 및 작업 분배            │
│  3. 결과 통합 및 사용자에게 응답              │
└────┬────────────┬────────────┬──────────────┘
     ↓            ↓            ↓
┌─────────┐  ┌─────────┐  ┌──────────────┐
│ Explore │  │  Plan   │  │general-purp. │
│ 에이전트 │  │ 에이전트 │  │   에이전트    │
│         │  │         │  │              │
│ auth/   │  │ 리팩토링 │  │ 코드 실제    │
│ 폴더    │  │ 계획    │  │ 수정 실행    │
│ 탐색    │  │ 수립    │  │              │
└────┬────┘  └────┬────┘  └──────┬───────┘
     │            │               │
     └────────────┴───────────────┘
                  ↓
          메인 에이전트로 결과 반환
                  ↓
          사용자에게 최종 보고
```

**실행 순서 예시**:
```
① Explore 에이전트: auth/ 폴더 구조 파악 (읽기, 병렬 가능)
② Plan 에이전트: 탐색 결과 기반 리팩토링 계획 수립 (읽기)
③ 메인 에이전트: 계획을 사용자에게 확인
④ general-purpose 에이전트: 승인된 계획 실행 (읽기+쓰기)
⑤ 메인 에이전트: 결과 통합 후 사용자에게 보고
```

## 실습: Explore 에이전트가 자동으로 쓰이는 상황 만들어보기

### 실습 1: 넓은 탐색 요청으로 Explore 트리거

```bash
claude

# 아래 요청들을 순서대로 입력해보세요

# 1. 전체 구조 탐색 (Explore 에이전트 자동 호출)
> "이 프로젝트의 전반적인 아키텍처를 설명해줘"

# Claude 응답 중 "[Explore 에이전트 실행 중...]" 같은
# 서브에이전트 활성화 메시지를 확인하세요

# 2. 특정 패턴 검색 (Explore 에이전트 자동 호출)
> "API 엔드포인트가 어디에 정의돼 있어?"

# 3. 비교 분석 (여러 Explore 에이전트 병렬 호출 가능)
> "프론트엔드와 백엔드 코드 구조를 비교해줘"
```

### 실습 2: Plan 에이전트 트리거

```bash
# Plan 에이전트가 호출되는 요청 예시
> "새 기능을 추가하기 전에 계획만 세워줘. 코드는 건드리지 마"
> "이 버그를 어떻게 고칠지 방법만 알려줘"
```

### 실습 3: 서브에이전트 실행 관찰

```bash
# --verbose 플래그로 서브에이전트 실행 과정 관찰
claude --verbose

> "전체 코드베이스에서 TODO 주석을 모두 찾아줘"

# 출력 예시:
# [Sub-agent: Explore] Starting file search...
# [Sub-agent: Explore] Found 47 files
# [Sub-agent: Explore] Scanning for TODO patterns...
# [Sub-agent: Explore] Complete: 23 TODOs found
#
# 결과: (메인 에이전트가 요약해서 제공)
```
