# 9-2. 커스텀 에이전트 (Custom Agents)

내장 에이전트 외에도 팀과 프로젝트의 특성에 맞는 **커스텀 에이전트**를 직접 정의할 수 있습니다. 커스텀 에이전트는 특정 역할(코드 리뷰어, 테스트 작성자 등)에 최적화된 시스템 프롬프트와 도구 제한을 가진 전용 에이전트입니다.

---

## 에이전트 파일 위치

### 프로젝트 전용 에이전트

```
프로젝트 루트/
└── .claude/
    └── agents/
        ├── code-reviewer.md
        ├── test-writer.md
        └── doc-generator.md
```

이 위치의 에이전트는 해당 프로젝트에서만 사용 가능합니다. 프로젝트 저장소에 커밋하면 팀원 전체가 공유할 수 있습니다.

### 전역 에이전트 (모든 프로젝트에서 사용)

```
~/.claude/
└── agents/
    ├── code-reviewer.md
    └── my-helper.md
```

홈 디렉토리의 `~/.claude/agents/`에 저장된 에이전트는 어느 프로젝트에서나 사용할 수 있습니다.

**우선순위**: 프로젝트 에이전트 > 전역 에이전트 (같은 이름이면 프로젝트 에이전트 우선)

---

## 에이전트 파일 형식

에이전트 정의 파일은 **Markdown(.md)** 형식입니다. 파일명이 에이전트의 이름이 됩니다.

- `code-reviewer.md` → `@code-reviewer` 로 호출
- `test-writer.md` → `@test-writer` 로 호출
- `my-agent.md` → `@my-agent` 로 호출

파일은 **YAML frontmatter + 시스템 프롬프트** 구조로 작성합니다.

```
---
(frontmatter: 에이전트 설정)
---

(시스템 프롬프트: 에이전트 역할과 행동 지침)
```

---

## 에이전트 파일 구조 예시

### 기본 예시: `code-reviewer.md`

```markdown
---
name: code-reviewer
description: 코드 품질, 보안, 성능을 검토하는 시니어 개발자 에이전트
model: sonnet
tools: Read, Grep, Glob
permissionMode: plan
maxTurns: 20
---

당신은 10년 경력의 시니어 풀스택 개발자입니다.
코드 리뷰 시 다음 항목을 반드시 확인하세요:

1. 보안 취약점 (SQL 인젝션, XSS, 인증 누락 등)
2. 성능 이슈 (N+1 쿼리, 불필요한 반복 등)
3. 코드 가독성과 유지보수성
4. 에러 처리 누락
5. 테스트 커버리지

발견한 문제는 심각도(Critical/Major/Minor)와 함께 보고하세요.
각 문제에 대해 개선 코드 예시를 함께 제공하세요.
```

---

## frontmatter 필드 완전 가이드

### `name` — 에이전트 이름

```yaml
name: code-reviewer
```

- 에이전트를 호출할 때 사용하는 식별자
- 파일명과 일치시키는 것을 권장 (혼란 방지)
- 하이픈(-) 사용 가능, 공백 불가

---

### `description` — 에이전트 설명 (자동 위임의 핵심)

```yaml
description: PR 코드의 품질, 보안 취약점, 성능 문제를 검토하는 시니어 개발자
```

- Claude가 **어떤 요청을 이 에이전트에게 위임할지** 판단하는 근거
- 구체적으로 작성할수록 자동 위임 정확도가 높아짐
- 에이전트가 잘하는 것과 못하는 것을 명시하면 더 좋음

**나쁜 예** (너무 모호):
```yaml
description: 도움이 되는 에이전트
```

**좋은 예** (구체적):
```yaml
description: >
  TypeScript/Python 코드의 보안 취약점(SQL 인젝션, XSS, CSRF),
  성능 이슈(N+1 쿼리, 메모리 누수), 코드 품질 문제를 분석하고
  심각도와 수정 방법을 제시하는 시니어 개발자 에이전트.
  파일 수정은 하지 않고 리뷰 보고서만 작성.
```

---

### `model` — 사용할 모델

```yaml
model: haiku    # 빠르고 저렴, 단순 작업용
model: sonnet   # 균형잡힌 능력, 일반 개발용 (기본값)
model: opus     # 가장 강력, 복잡한 설계/추론용
```

---

### `tools` — 허용 도구 목록 (화이트리스트)

```yaml
tools: Read, Grep, Glob
```

명시된 도구만 사용 가능합니다. 나머지는 모두 차단됩니다.

사용 가능한 도구 목록:
- `Read` — 파일 읽기
- `Write` — 파일 쓰기 (새 파일 생성)
- `Edit` — 파일 수정
- `Bash` — 셸 명령 실행
- `Grep` — 파일 내용 검색
- `Glob` — 파일 패턴 검색
- `WebFetch` — 웹 페이지 가져오기
- `WebSearch` — 웹 검색

---

### `disallowedTools` — 금지 도구 목록 (블랙리스트)

```yaml
disallowedTools: Write, Edit, Bash
```

명시된 도구는 사용 불가, 나머지는 모두 사용 가능합니다.

---

### `permissionMode` — 권한 모드

```yaml
permissionMode: default        # 위험 작업 시 매번 승인 요청
permissionMode: acceptEdits    # 파일 편집 자동 승인
permissionMode: plan           # 읽기 전용, 쓰기 불가
permissionMode: bypassPermissions  # 모든 승인 생략
```

---

### `maxTurns` — 최대 대화 턴 수

```yaml
maxTurns: 20
```

에이전트가 이 턴 수를 초과하면 자동으로 종료됩니다.
- 무한 루프 방지
- API 비용 제어
- 일반 작업: 10~20, 복잡한 작업: 50~100

---

### `memory` — 메모리 접근 범위

```yaml
memory: user      # 사용자 전체 메모리
memory: project   # 현재 프로젝트 메모리
memory: local     # 로컬 세션 메모리만
```

## 실용적인 에이전트 예제 3가지

### 예제 1: `code-reviewer.md` — 코드 리뷰

```markdown
---
name: code-reviewer
description: >
  변경된 코드의 품질, 보안, 성능을 검토하고 리뷰 보고서를 작성하는 에이전트.
  보안 취약점(인젝션, XSS 등), 성능 이슈(N+1 쿼리 등), 에러 처리,
  코드 가독성을 분석. 파일 수정 없이 보고서만 출력.
model: sonnet
tools: Read, Grep, Glob
permissionMode: plan
maxTurns: 30
---

당신은 10년 경력의 시니어 풀스택 개발자로, 코드 리뷰 전문가입니다.

## 리뷰 체크리스트

### 보안 (Critical 우선)
- [ ] SQL 인젝션 취약점
- [ ] XSS 취약점
- [ ] 인증/인가 누락
- [ ] 민감 정보 노출 (비밀번호, API 키 하드코딩)
- [ ] CSRF 취약점

### 성능
- [ ] N+1 쿼리 문제
- [ ] 불필요한 루프/중첩
- [ ] 큰 데이터셋 메모리 로딩
- [ ] 비동기 처리 누락

### 코드 품질
- [ ] 에러 처리 누락
- [ ] 중복 코드 (DRY 원칙 위반)
- [ ] 함수/변수 명명 불명확
- [ ] 복잡도 과다 (함수 30줄 초과 등)

## 보고서 형식

각 문제를 아래 형식으로 보고하세요:

**[심각도] 파일명:라인번호 — 문제 제목**
- 문제: (현재 코드의 문제점)
- 위험: (이 문제가 초래할 수 있는 결과)
- 수정: (개선된 코드 예시)
```

### 예제 2: `test-writer.md` — 테스트 작성

```markdown
---
name: test-writer
description: >
  구현 코드를 분석하여 단위 테스트, 통합 테스트를 작성하는 에이전트.
  Jest, pytest, vitest 등 프로젝트에 맞는 테스트 프레임워크 자동 감지.
  엣지 케이스와 에러 케이스를 포함한 포괄적인 테스트 작성.
model: sonnet
tools: Read, Grep, Glob, Write
permissionMode: acceptEdits
maxTurns: 40
---

당신은 TDD(테스트 주도 개발) 전문가입니다.

## 테스트 작성 원칙

1. **AAA 패턴** 사용: Arrange → Act → Assert
2. 테스트명은 "어떤 상황에서 무엇을 하면 어떤 결과" 형식
3. 각 테스트는 하나의 동작만 검증
4. 엣지 케이스 필수 포함:
   - null/undefined 입력
   - 빈 배열/객체
   - 최댓값/최솟값
   - 비정상 입력

## 작업 순서

1. 대상 파일의 모든 함수/메서드 파악
2. 각 함수의 입출력 분석
3. 프레임워크 확인 (package.json 또는 requirements.txt)
4. 테스트 파일 생성 (같은 폴더 또는 __tests__/ 폴더)
5. 정상 케이스 → 엣지 케이스 → 에러 케이스 순서로 작성

테스트 파일명: `[원본파일명].test.ts` 또는 `test_[원본파일명].py`
```

### 예제 3: `doc-generator.md` — 문서화

```markdown
---
name: doc-generator
description: >
  소스 코드를 분석하여 API 문서, 함수 설명, README를 자동 생성하는 에이전트.
  JSDoc, Python docstring 형식으로 인라인 주석 추가 및 Markdown 문서 생성.
model: sonnet
tools: Read, Grep, Glob, Write, Edit
permissionMode: acceptEdits
maxTurns: 50
---

당신은 기술 문서 작성 전문가입니다.

## 문서화 작업 유형

### 1. 인라인 주석 추가
- 함수/메서드에 JSDoc 또는 docstring 추가
- 복잡한 로직에 설명 주석 추가
- 파라미터, 반환값, 예외 상황 문서화

### 2. API 문서 생성
- 각 엔드포인트의 URL, 메서드, 파라미터, 응답 형식 문서화
- 요청/응답 예시 포함
- 에러 코드 및 처리 방법 명시

### 3. README 생성/업데이트
- 프로젝트 개요 및 기능 설명
- 설치 및 실행 방법
- 환경 변수 목록
- 기여 방법

## 작성 원칙

- 간결하고 명확하게 작성 (불필요한 내용 제거)
- 코드 예시는 실제 동작하는 예시 사용
- 한국어 또는 영어 (프로젝트에 맞게 통일)
```

## 실습: code-reviewer 에이전트 직접 만들기

### 1단계: 디렉토리 생성

```bash
# 프로젝트 루트에서 실행
mkdir -p .claude/agents
```

### 2단계: 에이전트 파일 작성

```bash
# .claude/agents/code-reviewer.md 파일 생성
cat > .claude/agents/code-reviewer.md << 'EOF'
---
name: code-reviewer
description: 코드 품질, 보안, 성능을 검토하는 시니어 개발자 에이전트
model: sonnet
tools: Read, Grep, Glob
permissionMode: plan
maxTurns: 20
---

당신은 시니어 풀스택 개발자입니다.
아래 항목을 확인하고 심각도(Critical/Major/Minor)와 함께 보고하세요:

1. 보안 취약점 (SQL 인젝션, XSS, 인증 누락)
2. 성능 이슈 (N+1 쿼리, 불필요한 반복)
3. 에러 처리 누락
4. 코드 가독성 문제

각 문제에 개선 코드 예시를 포함하세요.
EOF
```

### 3단계: 에이전트 확인

```bash
# Claude Code에서 에이전트 목록 확인
claude
> /agents
# 또는
> @code-reviewer 안녕? 네가 할 수 있는 일을 설명해줘
```

### 4단계: 에이전트로 실제 리뷰 요청

```bash
# 특정 파일 리뷰
> @code-reviewer src/auth/login.ts 파일을 리뷰해줘

# 폴더 전체 리뷰
> @code-reviewer src/api/ 폴더의 모든 파일을 리뷰해줘
```
