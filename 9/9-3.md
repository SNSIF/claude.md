# 9-3. 에이전트 옵션 (Agent Options)

커스텀 에이전트의 frontmatter에 설정할 수 있는 모든 옵션을 상세히 다룹니다. 옵션을 잘 조합하면 각 상황에 최적화된 에이전트를 만들 수 있습니다.

---

## 도구 제한: 화이트리스트 방식

허용할 도구만 명시하고 나머지는 모두 차단합니다.

```yaml
tools: Read, Grep, Glob
```

이 설정이 적용된 에이전트는 Read, Grep, Glob만 사용할 수 있습니다. Write, Edit, Bash 등은 자동으로 차단됩니다.

**사용 가능한 도구 전체 목록**:

| 도구 | 설명 |
|------|------|
| `Read` | 파일 내용 읽기 |
| `Write` | 새 파일 생성/전체 덮어쓰기 |
| `Edit` | 파일 일부 수정 |
| `Bash` | 셸 명령 실행 |
| `Grep` | 파일 내 패턴 검색 |
| `Glob` | 파일 이름 패턴 검색 |
| `WebFetch` | URL에서 웹 페이지 가져오기 |
| `WebSearch` | 웹 검색 |
| `TodoWrite` | 할 일 목록 관리 |

**화이트리스트 예시**:
```yaml
# 읽기 전용 에이전트
tools: Read, Grep, Glob

# 파일 생성/수정 포함
tools: Read, Write, Edit, Grep, Glob

# 명령 실행 포함
tools: Read, Write, Edit, Bash, Grep, Glob

# 웹 검색 포함
tools: Read, Grep, Glob, WebFetch, WebSearch
```

## 도구 금지: 블랙리스트 방식

금지할 도구만 명시하고 나머지는 모두 허용합니다.

```yaml
disallowedTools: Write, Edit, Bash
```

이 설정은 "모든 도구를 쓸 수 있지만 Write, Edit, Bash만 금지"를 의미합니다.

**블랙리스트 예시**:
```yaml
# 파일 수정과 명령 실행만 금지
disallowedTools: Write, Edit, Bash

# 외부 접근 금지
disallowedTools: WebFetch, WebSearch

# 파일 생성 금지 (읽기/수정은 허용)
disallowedTools: Write, Bash
```

---

## 화이트리스트 vs 블랙리스트 우선순위

`tools`와 `disallowedTools`를 동시에 설정하면 **화이트리스트(`tools`)가 우선** 적용됩니다.

```yaml
tools: Read, Grep, Glob, Write
disallowedTools: Write  # 이 설정은 무시됨
```

결과: Read, Grep, Glob, Write 모두 허용 (disallowedTools 무시)

**권장 사항**: 둘 중 하나만 사용하세요. 명확성을 위해 `tools`(화이트리스트) 방식을 권장합니다.

## 권한 모드 설정

에이전트마다 다른 권한 모드를 부여할 수 있습니다.

```yaml
permissionMode: plan           # 읽기 전용, 파일 수정/명령 실행 불가
permissionMode: default        # 위험 작업 시 사용자 승인 요청
permissionMode: acceptEdits    # 파일 편집 자동 승인, 나머지는 승인 요청
permissionMode: bypassPermissions  # 모든 승인 생략 (CI/자동화용)
```

**상황별 권장 권한 모드**:

```yaml
# 코드 리뷰 에이전트 (읽기만)
permissionMode: plan

# 일반 개발 에이전트 (편집은 자동, 명령은 확인)
permissionMode: acceptEdits

# CI 자동화 에이전트 (모든 작업 자동)
permissionMode: bypassPermissions

# 민감한 작업 에이전트 (매번 확인)
permissionMode: default
```

## 최대 턴 제한

에이전트가 처리할 수 있는 최대 대화 턴 수를 제한합니다.

```yaml
maxTurns: 10  # 10턴 초과 시 자동 종료
```

**maxTurns의 중요성**:

1. **무한 루프 방지**: 에이전트가 반복 작업에 갇히는 상황 방지
2. **비용 제어**: 예상치 못한 과도한 API 호출 방지
3. **작업 범위 제한**: 너무 광범위한 작업 방지

**상황별 권장 maxTurns**:

| 작업 유형 | 권장 maxTurns |
|-----------|--------------|
| 단순 파일 탐색 | 5~10 |
| 코드 리뷰 | 15~25 |
| 일반 개발 작업 | 30~50 |
| 대규모 리팩토링 | 50~100 |
| 복잡한 신규 기능 구현 | 100~200 |

## 모델 선택 전략 (비용 vs 능력)

```yaml
model: haiku   # 빠른 탐색/단순 작업
model: sonnet  # 일반 개발 작업 (기본값)
model: opus    # 복잡한 설계/추론
```

**상세 비교**:

| 모델 | 속도 | 비용 | 능력 | 적합한 작업 |
|------|------|------|------|-------------|
| haiku | 가장 빠름 | 저렴 | 기본 | 파일 탐색, 간단한 검색, 반복 작업 |
| sonnet | 보통 | 중간 | 우수 | 코드 작성, 리뷰, 일반 개발 |
| opus | 느림 | 비쌈 | 최강 | 아키텍처 설계, 복잡한 추론, 보안 분석 |

**비용 최적화 예시**:
```yaml
# 단순 파일 탐색 에이전트 → haiku로 비용 절감
name: file-explorer
model: haiku
tools: Read, Grep, Glob

# 복잡한 보안 분석 에이전트 → opus로 정확도 향상
name: security-auditor
model: opus
tools: Read, Grep, Glob
```

## 메모리 설정

```yaml
memory: project  # 현재 프로젝트 범위 메모리 접근
memory: user     # 사용자 전체 메모리 접근
memory: local    # 현재 세션만 (기본값)
```

## 스킬 주입

에이전트가 특정 스킬의 컨텍스트를 미리 로드하도록 설정합니다.

```yaml
skills:
  - api-conventions
  - error-handling
```

스킬이 주입되면 에이전트는 해당 스킬의 가이드라인과 패턴을 사전에 인지한 상태로 작업을 시작합니다.

**실용 예시**:
```yaml
# REST API 작성 에이전트에 API 관련 스킬 주입
skills:
  - api-conventions    # REST 설계 원칙
  - error-handling     # 에러 처리 패턴
  - authentication     # 인증 구현 패턴
```

## 훅 설정 (에이전트 내부 훅)

에이전트 파일 내부에서도 훅을 정의할 수 있습니다. 에이전트가 특정 도구를 사용하기 전/후에 동작을 추가합니다.

```yaml
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "echo '명령 실행: $TOOL_INPUT'"
  PostToolUse:
    - matcher: "Write"
      hooks:
        - type: command
          command: "echo '파일 생성됨: $TOOL_RESULT'"
```

**실용적인 훅 예시**:
```yaml
hooks:
  # Bash 명령 실행 전 로그 기록
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "echo '[$(date)] Bash 실행: $TOOL_INPUT' >> agent.log"

  # 파일 수정 후 git status 확인
  PostToolUse:
    - matcher: "Edit|Write"
      hooks:
        - type: command
          command: "git diff --stat"
```

## 전체 옵션을 조합한 실전 에이전트 예제

### 보안 감사 에이전트 (`security-auditor.md`)

```markdown
---
name: security-auditor
description: >
  코드베이스의 보안 취약점을 심층 분석하는 전문 보안 감사 에이전트.
  OWASP Top 10 기준으로 SQL 인젝션, XSS, CSRF, 인증 취약점,
  민감 정보 노출을 탐지. 파일 수정 없이 감사 보고서만 출력.
model: opus
tools: Read, Grep, Glob
permissionMode: plan
maxTurns: 50
memory: project
skills:
  - security-patterns
  - owasp-guidelines
---

당신은 OWASP 인증 보안 전문가입니다.

## 감사 범위

### OWASP Top 10 점검 항목
1. **A01 - 접근 제어 취약점**: 인가 없는 리소스 접근 가능 여부
2. **A02 - 암호화 실패**: 민감 데이터 평문 저장, 약한 암호화
3. **A03 - 인젝션**: SQL, NoSQL, OS, LDAP 인젝션
4. **A04 - 불안전한 설계**: 보안을 고려하지 않은 설계 패턴
5. **A05 - 잘못된 보안 설정**: 기본값 사용, 불필요한 기능 활성화
6. **A06 - 취약한 구성 요소**: 알려진 취약점이 있는 라이브러리
7. **A07 - 인증 및 세션 관리 실패**: 약한 비밀번호, 세션 하이재킹
8. **A08 - 소프트웨어 무결성 실패**: 검증 없는 외부 코드 사용
9. **A09 - 로깅 및 모니터링 실패**: 보안 이벤트 미기록
10. **A10 - SSRF**: 서버에서 조작된 요청 전송

## 보고서 형식

### 요약
- 총 발견 취약점: N개 (Critical: X, High: Y, Medium: Z, Low: W)

### 상세 항목
**[CRITICAL] 파일:라인 — 취약점 제목**
- OWASP 분류: A0X
- 설명: 취약점 상세 설명
- 영향: 악용 시 발생 가능한 피해
- 재현: 공격 시나리오 (선택)
- 권장 조치: 수정 방법 및 코드 예시

감사 완료 후 전체 보고서를 `security-audit-report.md`로 저장을 권고하세요.
```

## 실습: 옵션을 바꿔가며 에이전트 동작 비교

### 실습 1: 모델 변경으로 속도/비용 비교

```bash
# haiku 모델 에이전트
cat > .claude/agents/fast-explorer.md << 'EOF'
---
name: fast-explorer
description: 빠른 코드베이스 탐색을 위한 경량 에이전트
model: haiku
tools: Read, Grep, Glob
permissionMode: plan
maxTurns: 10
---
코드베이스를 빠르게 탐색하고 파일 구조를 요약하세요.
EOF

# sonnet 모델 에이전트
cat > .claude/agents/smart-explorer.md << 'EOF'
---
name: smart-explorer
description: 깊이 있는 코드 분석을 위한 스마트 탐색 에이전트
model: sonnet
tools: Read, Grep, Glob
permissionMode: plan
maxTurns: 20
---
코드베이스를 깊이 있게 분석하고 아키텍처, 패턴, 잠재적 문제를 파악하세요.
EOF
```

```bash
# Claude에서 두 에이전트 응답 비교
claude
> @fast-explorer 이 프로젝트 구조 설명해줘
# → 빠르고 간략한 답변

> @smart-explorer 이 프로젝트 구조 설명해줘
# → 더 상세하고 심층적인 답변
```

### 실습 2: maxTurns로 작업 범위 제한 확인

```bash
cat > .claude/agents/limited-agent.md << 'EOF'
---
name: limited-agent
description: 테스트용 턴 제한 에이전트
model: sonnet
tools: Read, Grep, Glob
maxTurns: 3
---
주어진 작업을 수행하세요.
EOF
```

```bash
claude
> @limited-agent 전체 프로젝트에서 모든 함수를 찾아서 목록으로 만들어줘
# → 3턴 후 자동 종료되어 완전한 결과를 못 얻는 것을 확인
```

### 실습 3: 권한 모드별 동작 차이 확인

```bash
# plan 모드 에이전트 (파일 수정 불가)
cat > .claude/agents/readonly-agent.md << 'EOF'
---
name: readonly-agent
description: 읽기 전용 테스트 에이전트
model: sonnet
tools: Read, Write, Edit
permissionMode: plan
---
파일을 분석하세요.
EOF

# acceptEdits 모드 에이전트 (파일 수정 가능)
cat > .claude/agents/edit-agent.md << 'EOF'
---
name: edit-agent
description: 편집 자동 승인 테스트 에이전트
model: sonnet
tools: Read, Write, Edit
permissionMode: acceptEdits
---
파일을 분석하고 필요시 수정하세요.
EOF
```

```bash
claude
# plan 모드 — 파일 수정 시도 시 차단됨
> @readonly-agent test.txt 파일에 "Hello" 한 줄 추가해줘
# → "plan 모드에서는 파일을 수정할 수 없습니다" 메시지

# acceptEdits 모드 — 파일 수정 자동 승인
> @edit-agent test.txt 파일에 "Hello" 한 줄 추가해줘
# → 승인 요청 없이 바로 수정 완료
```
