# 11-2. Extended Thinking

## Extended Thinking 개념

Extended Thinking(확장 사고)은 Claude가 최종 응답을 생성하기 전에 문제를 단계별로 깊이 분석하는 모드입니다. 마치 사람이 어려운 문제를 종이에 메모하며 생각하듯, Claude도 내부적으로 추론 과정을 거칩니다. 이 사고 과정은 사용자에게 표시되기도 하며(투명성), 최종 답변의 품질을 크게 향상시킵니다.

---

## 일반 응답 vs 확장 사고 비교

| 항목 | 일반 응답 | 확장 사고 |
|------|-----------|-----------|
| 응답 속도 | 빠름 | 느림 (수초~수십 초) |
| 답변 품질 | 표준 | 향상됨 |
| 토큰 비용 | 적음 | 많음 |
| 적합한 작업 | 간단한 질의, 코드 작성 | 복잡한 분석, 설계, 디버깅 |
| 사고 과정 표시 | 없음 | 옵션에 따라 표시 |

**예시**: 복잡한 알고리즘 설계

```
일반 응답:
"이진 탐색 트리를 사용하면 O(log n) 복잡도로 해결할 수 있습니다."

확장 사고:
[내부 사고 과정]
- 문제 분석: 데이터 크기, 쿼리 패턴, 업데이트 빈도 확인
- 자료구조 비교: BST vs Skip List vs B-Tree 장단점
- 메모리/시간 트레이드오프 계산
- 캐시 효율성 고려
- 실제 운영 환경 조건 검토

[최종 응답]
"데이터 크기가 수백만 건이고 범위 쿼리가 빈번한 경우라면,
 B-Tree 기반 인덱스가 더 적합합니다. 이유는..."
```

---

## 활성화 방법

### 방법 1: 기본 활성화 (자동)

Claude는 질문의 복잡도를 자동으로 판단하여 필요시 확장 사고를 활성화합니다.

```bash
# 단순 질문 → 일반 응답
claude -p "파이썬에서 리스트 정렬 방법은?"

# 복잡한 질문 → 자동으로 확장 사고 사용
claude -p "분산 시스템에서 CAP 정리를 고려한 데이터베이스 설계를 제안해줘"
```

### 방법 2: 단축키 토글

```
Alt+T (Windows/Linux)
Option+T (macOS)
```

Claude Code 패널 또는 터미널에서 응답 중에도 토글 가능합니다.

### 방법 3: 프롬프트에 키워드 포함

특수 키워드를 프롬프트에 포함하면 확장 사고가 강제 활성화됩니다:

```bash
# "ultrathink" 키워드 사용
claude -p "ultrathink: 이 마이크로서비스 아키텍처의 잠재적 장애 지점을 분석해줘"

# 또는 문장 안에 포함
claude -p "이 알고리즘을 ultrathink로 최적화해줘"

# "think hard", "think step by step" 도 유사하게 동작
claude -p "이 버그를 think hard로 분석해줘"
```

### 방법 4: `/effort` 커맨드로 수준 조정

```bash
# 낮은 노력
/effort low

# 기본값 (medium)
/effort medium

# 높은 노력
/effort high

# 최대 노력 (Opus 모델 전용)
/effort max
```

---

## 노력 수준 (Effort Level) 상세

### `low` — 빠른 응답

```bash
/effort low
# 또는
claude --effort low -p "질문"
```

- 사고 토큰: 최소 (수백 토큰)
- 응답 시간: 거의 즉각
- 적합한 경우: 단순 코드 수정, 짧은 질의응답

### `medium` — 기본값

```bash
/effort medium
```

- 사고 토큰: 보통 (수천 토큰)
- 응답 시간: 수 초
- 적합한 경우: 일반적인 개발 작업, 코드 리뷰

### `high` — 깊은 사고

```bash
/effort high
# 또는
claude --effort high -p "질문"
```

- 사고 토큰: 많음 (수만 토큰)
- 응답 시간: 10-30초
- 적합한 경우: 복잡한 버그 추적, 설계 검토

### `max` — 최대 사고 (Opus 전용)

```bash
/effort max
# claude-opus-4-6 모델에서만 사용 가능
```

- 사고 토큰: 최대 (최대 64K 토큰)
- 응답 시간: 30초~수 분
- 적합한 경우: 매우 복잡한 알고리즘, 전체 시스템 아키텍처 설계

---

## `MAX_THINKING_TOKENS` 환경 변수

사고에 사용할 최대 토큰 수를 제한하여 비용을 관리합니다.

```bash
# 환경 변수 설정
export MAX_THINKING_TOKENS=8000   # 8K 토큰으로 제한
export MAX_THINKING_TOKENS=32000  # 32K 토큰 허용

# 단일 명령에 적용
MAX_THINKING_TOKENS=16000 claude -p "ultrathink: 시스템 설계해줘"
```

```json
// ~/.claude/settings.json
{
  "maxThinkingTokens": 16000,
  "defaultEffort": "medium"
}
```

**토큰 예산 가이드**:

| 토큰 범위 | 사용 사례 |
|-----------|-----------|
| 1K - 4K | 간단한 로직 분석 |
| 4K - 16K | 복잡한 버그 디버깅 |
| 16K - 32K | 아키텍처 설계 |
| 32K+ | 대규모 시스템 분석 (Opus 권장) |

## 언제 Extended Thinking을 사용할까

### 적합한 경우

**1. 복잡한 알고리즘 설계**
```
> ultrathink: 실시간 협업 편집기에서
  여러 사용자가 동시에 같은 문서를 편집할 때
  충돌을 해결하는 알고리즘을 설계해줘
```

**2. 버그 원인 분석**
```
> /effort high
> 이 코드가 멀티스레드 환경에서만 크래시가 발생해.
  스택 트레이스는 다음과 같아: [스택 트레이스 첨부]
  원인이 뭔지 깊이 분석해줘
```

**3. 아키텍처 결정**
```
> ultrathink: 현재 모놀리식 아키텍처를 마이크로서비스로
  전환할 때의 장단점, 리스크, 마이그레이션 전략을 분석해줘.
  트래픽은 일평균 100만 요청이야.
```

**4. 보안 취약점 분석**
```
> /effort high
> @src/api/ 폴더 전체를 OWASP Top 10 기준으로
  보안 취약점을 철저히 분석해줘
```

### 부적합한 경우 (일반 응답이 더 효율적)

```
# 단순 변환 작업 → 확장 사고 불필요
claude -p "이 JSON을 CSV로 변환해줘"

# 간단한 코드 생성 → 확장 사고 불필요
claude -p "파이썬 파일 읽기 함수 작성해줘"

# 정보 검색 → 확장 사고 불필요
claude -p "React useEffect 사용법 알려줘"
```

## 비용 고려사항

```
모델별 토큰 비용 (참고용, 실제 가격은 공식 문서 확인):

claude-opus-4-6:
  Input:  $15 / 1M tokens
  Output: $75 / 1M tokens
  Thinking tokens도 output에 포함

claude-sonnet-4-6:
  Input:  $3 / 1M tokens
  Output: $15 / 1M tokens

claude-haiku-4-5:
  Input:  $0.25 / 1M tokens
  Output: $1.25 / 1M tokens
```

**비용 절감 전략**:

```bash
# 1. 적절한 모델 선택
# 대부분의 작업: sonnet
# 최고 복잡도만: opus

# 2. 토큰 예산 제한
MAX_THINKING_TOKENS=8000 claude -p "ultrathink: 분석해줘"

# 3. effort 수준 적절히 설정
/effort medium  # 기본값이 대부분의 경우에 충분

# 4. 캐시 활용
# 자주 사용하는 시스템 프롬프트는 prompt caching 활용
```

## 실습

### 실습 1: 노력 수준별 응답 품질 비교

동일한 질문을 세 가지 수준으로 질문하여 차이를 비교합니다.

```bash
# 1단계: low effort
claude -p "이진 탐색 알고리즘의 시간복잡도 분석"
# (응답 시간과 내용 메모)

# 2단계: high effort
/effort high
claude -p "이진 탐색 알고리즘의 시간복잡도 분석"
# (응답 시간과 내용 메모)

# 3단계: ultrathink
claude -p "ultrathink: 이진 탐색 알고리즘의 시간복잡도 분석"
# (응답 시간과 내용 메모)
```

**비교 포인트**:
- 고려된 엣지 케이스 수
- 분석의 깊이
- 추가로 언급된 관련 사항
- 응답 생성 시간

### 실습 2: 실제 버그 분석

```python
# bug_sample.py — 버그가 있는 코드
import threading
import time

counter = 0

def increment():
    global counter
    for _ in range(10000):
        temp = counter
        time.sleep(0.000001)  # 인위적 지연
        counter = temp + 1

threads = [threading.Thread(target=increment) for _ in range(5)]
for t in threads:
    t.start()
for t in threads:
    t.join()

print(f"예상: 50000, 실제: {counter}")
# 실제 결과가 50000보다 훨씬 작게 나옴
```

```bash
claude --effort high -p "
이 Python 코드에서 counter가 50000이 되어야 하는데
항상 더 작은 값이 나옵니다. 원인을 분석하고
해결책을 제시해줘. @bug_sample.py
"
```

### 실습 3: 아키텍처 설계

```bash
claude -p "ultrathink:
다음 요구사항을 가진 실시간 채팅 서비스의 백엔드 아키텍처를 설계해줘:
- 동시 접속자: 최대 100만 명
- 메시지 전송 지연: 100ms 이하
- 메시지 영속성 보장
- 이전 대화 이력 검색 기능
- 멀티 디바이스 동기화

기술 스택 선택 이유, 스케일링 전략, 장애 대응 방안을 포함해줘.
"
```
